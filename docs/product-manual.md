# 产品说明书

## 1. 产品基本信息

- 产品中文名：无限画布多模型聊天室
- 产品英文名：MeshCanvas
- 产品形态：本地部署的 Web 应用
- 技术形态：FastAPI 后端 + 原生 JavaScript 前端 + SQLite 本地数据库
- 核心能力：一条问题同时调用多个大模型，在同一张无限画布中流式展示、对比、分支和继续讨论

## 2. 产品定位

无限画布多模型聊天室是一款面向个人、团队和内部业务场景的多模型协作对话工具。它的设计目标不是做一个单轮聊天框，而是把多模型问答、联网搜索、讨论轮次、分支延展和结果沉淀放在同一张可操作的画布里，让用户从“问一次、看一个回答”升级为“问一次、看多个回答、继续推演、保留思路结构”。

该产品适合部署在个人电脑、内网服务器或轻量云主机上，连接任意 OpenAI 兼容或 Anthropic 格式的大模型网关，用于模型对比、内容生产、问题分析、研究整理、团队演示和内部知识辅助。

## 3. 目标用户

- 需要同时对比多个模型输出的高级用户
- 需要保留推理过程与分支结构的内容创作者
- 需要在本地或私有环境中部署 AI 工具的团队
- 需要对用户、模型、定价和用量进行统一管理的管理员
- 需要按积分、按模型成本管理 LLM 使用量的运营方

## 4. 核心价值

- 多模型并发：同一条问题可一次性发送给多个模型，减少切换成本
- 结果可视化：所有回答以节点方式落在画布上，便于横向比较和结构化整理
- 可继续推演：支持从任意节点继续对话、建立分支、进行多轮讨论
- 可引入外部信息：支持 Firecrawl 联网搜索，并可通过预处理模型自动判断是否需要搜索
- 可运营：自带管理后台、用户管理、模型配置、积分计费、用量统计和审计日志
- 可私有化：本地数据库、本地会话、服务端保存 API Key，不依赖第三方前端 SaaS

## 5. 典型使用场景

### 5.1 多模型回答对比

用户输入一个问题，同时选择多个模型。系统并发发起请求，每个模型独立流式返回内容，用户在画布上即时看到多路输出，并对结果进行横向比较。

### 5.2 研究与资料整理

用户开启联网搜索，让系统补充外部网页信息，再由多个模型分别进行总结、分析或写作。用户可以继续圈选节点、做总结、扩展新分支，形成一张完整的研究画布。

### 5.3 多轮讨论和观点碰撞

用户将讨论轮数设置为 2 到 4 轮，让多个模型在服务端进行多轮互相回应。系统在每一轮中把其他模型的观点注入当前模型上下文，形成更接近“多方讨论”的体验。

### 5.4 团队内部 AI 平台

管理员统一配置模型网关、联网搜索、模型价格、用户积分和注册开关，普通用户只负责使用。这样既能保持统一运营，又能控制成本和权限。

## 6. 产品总体架构

产品由三部分组成：

- 用户端：登录后进入无限画布，用于发起对话、查看模型回答、建立分支、切换画布和查看个人用量
- 管理后台：管理员登录后管理用户、模型、积分、系统配置和审计日志
- 服务端与存储：FastAPI 提供页面、接口和 WebSocket 服务；SQLite 保存业务数据；JSONL 保存请求日志

整体请求链路如下：

1. 用户在网页端发送问题
2. 前端通过 WebSocket 把请求发往服务端
3. 服务端为每个模型创建独立并发任务
4. 每个模型的流式输出通过同一 WebSocket 回推给前端
5. 前端把回答渲染为画布节点，并维护位置、连接关系和状态

## 7. 用户端功能说明

### 7.1 账号体系

- 支持注册、登录、退出登录
- 使用 HttpOnly Session Cookie 保存登录态
- 登录后方可访问画布、用量、对话和受保护接口
- 支持用户端独立会话，不与管理后台会话混用

### 7.2 首次安装与初始化向导

当系统尚未完成模型配置时，访问任意页面会自动跳转到 `/setup`：

- 配置全局 API 地址
- 选择接口格式：`openai` 或 `anthropic`
- 配置全局 API Key
- 配置可用模型列表

保存完成后，系统会：

- 写入模型配置文件
- 初始化数据库
- 自动确保默认管理员账号存在

### 7.3 无限画布

无限画布是产品最核心的交互区域，支持：

- 鼠标滚轮缩放
- 空格键配合拖拽平移画布
- 节点自由拖动
- 框选多个节点
- 小地图查看整体布局
- 一键回到当前对话区域
- 清空当前画布显示内容

每次对话请求会形成一个集群，包含用户消息节点和多个模型回复节点。不同请求集群可在同一张画布内并存。

### 7.4 多画布管理

用户可以管理多张画布：

- 查看画布列表
- 新建画布
- 切换画布
- 重命名画布
- 删除画布

系统会按画布维度保存对话状态和节点布局，让不同主题互不干扰。

### 7.5 多模型并发对话

产品支持同一次提问调用多个模型：

- 一个问题同时发给多个模型
- 每个模型独立流式返回，不互相阻塞
- 某个模型出错不会影响其它模型继续完成
- 前端实时显示每个模型的生成状态、完成状态和错误状态

### 7.6 多轮讨论

用户可在发送问题前设置讨论轮数：

- 1 轮：标准多模型并发回答
- 2 到 4 轮：模型之间进行多轮讨论

在多轮模式中，系统会把各模型上一轮输出整理后注入下一轮上下文，使模型在回答时参考其他模型观点，形成更深入的交叉讨论。

### 7.7 联网搜索

联网能力基于 Firecrawl 实现，用户可选择三种模式：

- 自动：由预处理模型判断是否需要联网
- 开：强制联网
- 关：不联网

联网流程支持：

- 搜索前预处理
- 搜索执行状态展示
- 搜索完成后结果注入
- 搜索结果整理后再喂给模型

如果管理员未配置 Firecrawl 或未配置可用于自动判断的预处理模型，对应能力会受到限制。

### 7.8 思考模式

用户可开启“思考”开关。开启后，提示词策略会更强调验证、分析和推理过程，适用于需要更谨慎、更结构化输出的场景。

### 7.9 分支对话

用户可从某个模型某一轮回答继续延展出新分支：

- 从已有回答继续追问
- 保留原始回答作为父节点
- 在同一张画布中形成新的推演路径

这一能力适合探索不同假设、不同方向和不同写作版本。

### 7.10 失败重试

当某个模型请求失败时，用户可以对单个模型发起重试，而不必重跑整组请求，节省时间和成本。

### 7.11 框选与圈选继续操作

用户可通过框选选中多个节点，并进行以下操作：

- 继续对话
- 建立分支
- 清除选择

这使得对话不再局限于单个节点，而可以从一组上下文继续延展。

### 7.12 圈选总结

系统支持对选中的节点生成总结：

- 汇总多个节点内容
- 形成精简摘要
- 辅助用户整理长对话、多模型结果或复杂讨论

### 7.13 对话分析与结题

产品支持对会话内容进行进一步分析：

- 对会话消息进行分析总结
- 对某个请求生成结题或结论
- 结题失败时可重试

这类能力适合在多轮对话完成后，把信息压缩为可执行结论。

### 7.14 账号与用量页面

用户可在 `/settings` 查看个人维度信息：

- 用户名
- 今日消耗
- 近 7 天消耗
- 历史总量
- 当前积分余额
- 逐条用量明细

用量明细可查看模型、输入 tokens、输出 tokens、响应时间和积分消耗，帮助用户理解自己的使用情况。

### 7.15 自定义 API Key

系统支持用户按模型设置自定义 API Key：

- 可在“使用积分”与“使用自定义 API Key”之间切换
- 可按模型填写 Key
- 保存后刷新画布即可生效
- 使用自定义 Key 时，模型调用可绕开平台积分计费逻辑

这一能力适合既想使用统一平台界面，又希望自带模型资源的用户。

## 8. 管理后台功能说明

管理后台入口为 `/admin`，使用独立 `admin_session` Cookie 鉴权。

### 8.1 管理员登录

- 独立于普通用户登录
- 默认管理员通常为 `admin / admin`
- 登录成功后进入控制台
- 后台登录与用户端登录互不影响

### 8.2 用户管理

管理员可以：

- 查看用户列表
- 查看用户余额、累计充值和累计消耗
- 为用户充值
- 为用户扣减积分
- 重置用户密码
- 修改用户角色

系统带有保护逻辑：

- 至少保留一名管理员
- 管理员不能移除自己的管理员身份

### 8.3 模型配置

管理员可统一配置平台使用的大模型连接参数：

- 全局 API 地址
- 全局 API Key
- API 格式：OpenAI 或 Anthropic
- 模型列表
- 预处理模型
- 用户自定义 API 的独立网关地址与格式
- 请求额外参数
- 请求头覆盖

此外还支持连通性测试，帮助管理员验证当前模型网关是否可用。

### 8.4 联网搜索配置

管理员可配置 Firecrawl：

- Firecrawl Key
- 国家代码
- 搜索超时

未配置 Firecrawl 时，联网搜索功能不可用。

### 8.5 模型定价

管理员可按模型配置积分价格：

- 输入 tokens 每千的积分单价
- 输出 tokens 每千的积分单价
- 可对模型列表中的模型快速设价
- 也可手动为任意模型 ID 设置定价

### 8.6 用量统计

管理员可查询平台整体或指定用户的使用情况：

- 按模型查看调用次数
- 查看输入 tokens、输出 tokens 和总 tokens
- 查看累计积分消耗
- 查看充值与扣减记录

### 8.7 系统配置

管理员可以设置系统级参数：

- 新用户默认赠送积分
- 低余额提醒阈值
- 每次搜索扣减积分
- 是否开放注册

这些配置直接影响平台运营策略和成本控制。

### 8.8 审计日志

后台记录关键管理操作，便于追踪责任和回溯变更，覆盖：

- 充值和扣减
- 角色变更
- 密码重置
- 定价修改
- 系统配置修改
- 模型连通性测试

## 9. 计费与运营逻辑

### 9.1 积分体系

系统采用积分制：

- 每个用户有独立余额
- 模型调用按模型价格扣分
- 搜索功能可单独扣分
- 管理员可以充值或扣减

### 9.2 模型成本核算

每次请求的输入和输出 Token 都会记录下来，并结合模型单价计算消耗。这样既能支撑用户端查看账单，也能支撑后台做运营和结算。

### 9.3 自定义 Key 绕开平台模型计费

当用户使用自定义 API Key 时，平台可不再对模型调用本身进行积分扣费。该机制有利于兼顾平台统一入口和用户自带资源。

## 10. 数据存储与持久化

### 10.1 SQLite

核心业务数据保存在 SQLite 中，包括：

- 用户与会话
- 对话请求与模型结果
- 请求事件
- 画布与节点位置
- 用户设置
- 模型定价
- 积分余额与流水
- 审计日志
- 请求摘要

### 10.2 日志

系统还会按天写入 JSONL 请求日志，便于排查和分析。

### 10.3 持久化范围说明

数据库保存的是请求记录、结果、配置和布局信息，但当前对话运行时上下文仍主要保存在 WebSocket 连接内存中。也就是说：

- 刷新页面后，正在进行中的会话状态会重置
- 服务重启后，内存中的线程上下文不会自动恢复

## 11. 安全机制

产品已实现多层基础安全能力：

- HttpOnly Cookie 登录态
- 密码哈希存储
- POST 与 WebSocket Origin 校验
- CSP 安全头
- `X-Frame-Options: DENY`
- 登录、注册、部分后台接口和 WebSocket 的限流
- 登录失败告警日志

这些能力用于降低常见的会话劫持、跨站请求、暴力尝试和前端注入风险。

## 12. 部署方式

### 12.1 标准部署

适合开发机、个人设备和轻量服务器：

1. 创建虚拟环境
2. 安装依赖
3. 准备模型配置
4. 启动 `uvicorn`

### 12.2 反向代理部署

若在 HTTPS 或反向代理后运行：

- 需正确传递 `X-Forwarded-Proto`
- 需为 `uvicorn` 配置可信代理
- 否则 Cookie 的 `Secure` 行为可能异常，影响后台登录和会话保持

## 13. 当前限制与边界

- 当前无自动化测试与完整 lint 体系
- 当前对话上下文主要存在于内存中，不支持服务重启后完整恢复
- SQLite 更适合单机和轻量并发，不适合多进程高并发写入
- 联网搜索依赖 Firecrawl，可用性受外部服务影响
- 自动搜索判断依赖预处理模型是否已配置

## 14. 适用价值总结

无限画布多模型聊天室的价值不只是“把多个模型放在一起”，而是把“并发回答、可视化比较、分支推演、联网检索、积分运营和后台管理”组合成一套完整的产品机制。它既能作为个人研究与写作工具，也能作为团队内部 AI 工作台和可运营的多模型平台原型。
